单页应用基础

理解 SPA 的加载流程、客户端路由以及它与传统多页网站的区别

SPA 是 Single-Page Application 的缩写，即单页应用。它描述的是一种应用组织和导航方式，并不表示整个产品只能有一个功能页面。

传统多页网站

传统多页网站访问不同地址时，服务器通常返回不同 HTML 文档：

访问 /products
   ↓
服务器返回商品列表 HTML

访问 /products/1
   ↓
服务器返回商品详情 HTML

每次页面跳转都可能重新加载完整文档、样式和脚本。

单页应用

单页应用首次加载基础 HTML 和 JavaScript 后，页面内容与路由切换可以由前端代码完成：

浏览器请求 index.html
        ↓
加载 JavaScript 模块
        ↓
React 创建组件界面
        ↓
路由决定显示哪个页面组件
        ↓
交互和数据变化更新局部界面

用户可以访问首页、列表页和详情页，但这些页面通常是由客户端路由切换的 React 组件。

SPA 不等于没有 URL

好的单页应用仍然应该有清晰 URL：

/
/products
/products/42
/cart
/account

URL 让用户可以刷新、收藏、分享和使用浏览器前进后退。后续会使用 React Router 学习客户端路由。

SPA 的优点

  • 页面切换时可以保留部分应用状态
  • 局部更新通常不需要重新加载整个文档
  • 适合交互密集的管理后台和 Web 应用
  • 前后端可以通过 API 明确分工
  • 多个页面可以复用相同布局和组件

SPA 需要额外处理的问题

客户端应用还要考虑：

  • 路由与导航
  • 数据请求和缓存
  • 加载、空数据与错误状态
  • 身份认证
  • 搜索引擎可见性
  • 首屏加载性能
  • 生产环境路由回退

React 本身不内置完整路由和数据请求方案，这些能力需要框架或其他工具配合。

路由回退为什么重要

在客户端从 /products 点击进入 /products/42 时，路由库可以直接切换组件。

但如果用户刷新 /products/42，浏览器会向服务器请求这个地址。服务器必须返回应用入口 index.html，再由客户端路由显示详情页。

如果服务器只寻找名为 products/42 的真实文件，就会返回 404。这就是 SPA 部署常见的路由回退问题。

SPA 与 React 的关系

React 可以用于 SPA，也可以用于：

  • 传统网站中的局部组件
  • 通过框架生成的静态页面
  • 服务端渲染页面
  • 原生移动应用

React 是 UI 库，SPA 是应用架构方式，两者不是同一个概念。

客户端项目与 React 框架

Vite 可以创建一个客户端 SPA 起点，但不会自动提供路由、服务端渲染和完整数据方案。

React 框架通常会把其中多项能力组合起来。为了先看清组件、Props、State 和 Effect，本课程从 Vite 客户端项目开始。

思考练习

判断下面功能是否适合使用 SPA，并说明原因：

  1. 只有联系方式的静态个人名片页
  2. 包含筛选、图表和实时更新的数据后台
  3. 具有购物车、账户和结算流程的电商应用
  4. 只有几篇固定文章的小型说明网站

答案不只取决于功能数量，还要考虑团队、性能、SEO 和维护成本。
